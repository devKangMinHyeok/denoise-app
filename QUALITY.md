# 품질 시스템

보이스 클로닝 품질을 "느낌"이 아니라 **숫자로** 지키기 위한 체계.

## 북극성 지표 (North Star): VCS

**VCS (Voice Clone Score, 0~100)** — 릴리스 간 품질 추이를 추적하는 단일 숫자.

```
VCS = 100 × ( 0.5 × min(SIM / 0.909, 1)    ← 목소리가 얼마나 "그 사람"인가
            + 0.3 × (1 − CER/100)           ← 대본을 얼마나 정확히 읽는가
            + 0.2 × MOS / 5 )               ← 얼마나 자연스러운가
```

- `0.909` 는 **본인 육성끼리 비교했을 때의 유사도 실측 기준선**. SIM 항이 1.0이면
  "본인이 딴 날 말한 것만큼 같은 목소리"라는 뜻이라, 그 이상은 점수를 더 주지 않는다
  (기준선 과최적화 방지).
- 가중치 근거: 클로닝의 존재 이유가 "그 목소리"(0.5) > "대본 정확성"(0.3) > "질감"(0.2).
- 현재 릴리스 실측: **약 91점** (SIM 0.917, CER 0%, MOS 3.50).

## 결과물 평가지표 3종 (`core/metrics.py`)

| 지표 | 뜻 | 측정 방법 | 게이트 |
|------|-----|----------|--------|
| SIM | 화자 유사도 (0~1) | 스피커 임베딩(Resemblyzer) 코사인 | ≥ 0.85 |
| CER | 글자 오류율 (%) | Whisper 받아쓰기 ↔ 대본 대조 (공백·부호 정규화) | ≤ 3% |
| MOS | 자연스러움 (1~5) | DNSMOS P.835 OVRL | ≥ 3.0 |
| VCS | 북극성 종합 | 위 3개 합성 | ≥ 85 |

게이트는 CI가 빠른 0.6B 모델 + 합성 픽스처로 돌기 때문에 실사용(1.7B) 실측보다
보수적으로 잡았다. 게이트 값은 `core/metrics.py`의 `GATES` 한 곳에서만 관리한다.

## CI 2단 구조 (`.github/workflows/`)

| 워크플로 | 트리거 | 러너 | 하는 일 | 소요 |
|----------|--------|------|---------|------|
| `ci.yml` | 모든 push/PR | ubuntu | 유닛 테스트 (필터 체인, CER 정규화, VCS 수식, 게이트 판정, API 계약, **아키텍처 경계**) — 모델 다운로드 없음 | ~1분 |
| `quality.yml` | 주 1회 + 수동 실행 | macos (Apple Silicon) | **실제 생성 → 채점 → 게이트** 스모크(대본 1개, `--limit 1`). 리포트 JSON 아티팩트 저장 | ~10분 (캐시 후) |
| 전체 게이트 | 릴리스 전 수동 | 로컬 맥 (또는 셀프호스티드) | 테스트 대본 **전체**로 `run_eval.py --fast` | ~2분 (M1 Pro) |

### 왜 2단 + 로컬 전체 게이트인가

- 코드 회귀(로직 깨짐)는 푸시마다 1분 안에 잡는다 — 무거운 모델 없이 순수 함수만.
- 품질 회귀(소리가 나빠짐)는 실제 생성 없이는 못 잡는다.
- 단, **GitHub 호스티드 맥 러너에는 GPU(Metal)가 없다.** 실측: 생성이 수십 배 느리고
  간헐적으로 멈춘다(무출력·행). 그래서 코어에 출력 검증 + 600초 타임아웃 + 재시도를
  넣었고, 호스티드에서는 대본 1개 스모크만 돌린다. 전체 3종 게이트는 GPU가 있는
  로컬/셀프호스티드에서 릴리스 전에 돌리는 것이 기준이다 (로컬 소요 ~2분).

### 테스트 자산

- `tests/fixtures/ref_fixture.wav` — **AI로 생성한 가상 목소리** (Qwen3-TTS 프리셋).
  실제 인물 목소리가 아니므로 공개 저장소에 안전. 절대 실제 사람 목소리로 교체하지 말 것.
- `quality/testset.txt` — 회귀 대본 세트. 일반 문장 + 숫자·영어 혼용 + 긴 호흡 문장.
  약점을 발견하면 여기에 문장을 추가하는 것이 곧 회귀 테스트 추가다.

## 로컬에서 돌려보기

```bash
pip install -r voice/requirements-voice.txt resemblyzer jiwer onnxruntime librosa "setuptools<81"
python3 quality/run_eval.py --fast            # CI와 동일
python3 quality/run_eval.py                   # 실사용(1.7B) 기준
python3 quality/run_eval.py --report r.json   # 리포트 저장
pytest tests -q                               # 유닛 테스트만
```

## 품질을 올리고 싶을 때의 절차

1. `quality/testset.txt`에 문제 문장을 추가한다 (재현 케이스 = 회귀 테스트).
2. 파이프라인/설정 후보를 만든다 (`core/clone.py`).
3. `run_eval.py`로 전후 비교 — **VCS가 오르고 게이트를 통과하면 채택**.
4. README의 실측표와 `GATES`(필요시)를 갱신하고 PR에 리포트를 첨부한다.
